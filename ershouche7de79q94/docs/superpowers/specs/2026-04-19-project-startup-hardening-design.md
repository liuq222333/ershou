# 项目完整启动方案设计

## 背景

当前项目的启动方式存在三类明显问题：

1. 历史文档和脚本已经过期，仍然指向旧端口和旧命令。
2. 后端依赖环境与代码实现存在版本兼容风险，导致项目在不同机器上容易出现“昨天能跑、今天起不来”的情况。
3. 项目由后端、管理端、用户端三部分组成，但缺少统一入口，日常启动依赖人工分别执行多个命令，容易漏掉某一端或使用错误环境。

用户希望把当前项目整理成“完整启动版”，做到一次执行即可把项目拉起，同时避免再次踩到 Flask 兼容、Node 版本、旧文档误导等问题。

## 目标

- 为项目提供统一、稳定、可重复的完整启动入口。
- 让后端、管理端、用户端都能被同一套脚本正确启动。
- 将当前真实可用的端口、命令、环境要求写入项目文档。
- 降低后续再次启动时报错的概率，尤其是 Python/Flask 与 Node 环境不一致导致的问题。
- 保持修复最小化，不引入新的复杂依赖或部署系统。

## 非目标

- 不在本次改造中引入 Docker、pm2、Supervisor 等额外进程管理方案。
- 不自动联网安装依赖，也不自动重建数据库。
- 不重构前后端业务代码，只围绕启动链路、文档和环境兼容做修复。
- 不处理生产部署，仅面向本地开发/演示环境。

## 当前现状

### 后端

- 当前真实可用启动命令为项目虚拟环境中的：
  - `.\.venv311\Scripts\python.exe .\run.py`
- 历史说明和 `run.bat` 仍然使用：
  - `python3 manage.py run`
- 当前代码中曾混入 Flask 2.x 的 `DefaultJSONProvider` 写法，但依赖实际固定在 `Flask==1.1.2`，导致后端启动失败。
- 端口实际为：
  - `8080`

### 管理端

- 位于：
  - `api/templates/front/admin`
- 启动命令为：
  - `npm run serve`
- 实际可用端口：
  - `8081`
- 当前 Node 环境已验证可用版本为：
  - `v16.20.2`

### 用户端

- 位于：
  - `api/templates/front/front`
- 启动命令为：
  - `npm run serve`
- 实际可用端口：
  - `8082`

### 文档和脚本问题

- `说明文档.md` 中仍写着旧地址：
  - 前台 `8123`
  - 后台 `8124`
- `run.bat` 仍指向旧命令，已经不能作为可靠入口。
- 项目没有统一的“启动全部/停止全部”脚本。

## 方案对比

### 方案 A：仅修正旧文档和旧脚本

只更新 `run.bat` 和文档，把过期命令改成当前能跑的命令。

优点：

- 改动最小。
- 风险低。

缺点：

- 仍需要手动分别启动三端。
- 无法提示环境问题和端口冲突。
- 日常使用仍然容易出错。

### 方案 B：完整启动版（采用）

新增统一启动脚本和停止脚本，统一检查环境、启动三端、输出地址和日志位置，同时修正文档和旧脚本。

优点：

- 启动路径统一。
- 使用成本最低。
- 可以吸收当前已经确认的环境约束，减少后续重复报错。

缺点：

- 会新增若干脚本文件。
- 需要维护脚本中的路径和端口信息。

### 方案 C：自愈式全自动启动

在完整启动版基础上，自动安装依赖、自动补环境、自动初始化数据库。

优点：

- 使用上最省心。

缺点：

- 对现有本地环境侵入性较强。
- 一旦自动修复出错，排查更复杂。
- 超出当前需求边界。

## 采用方案

采用方案 B：完整启动版。

核心原则：

- 统一入口。
- 显式检查。
- 日志可追踪。
- 与当前项目真实环境保持一致。
- 优先兼容现有依赖，而不是强推升级依赖版本。

## 设计方案

### 1. 统一启动入口

在项目根目录新增：

- `start-project.ps1`
- `stop-project.ps1`
- `start-project.bat`

并调整现有：

- `run.bat`

其中：

- `start-project.ps1` 负责完整启动逻辑，是主入口。
- `start-project.bat` 负责方便双击或命令行调用 PowerShell 脚本。
- `run.bat` 不再直接执行过期命令，而是转发到新的统一启动入口。
- `stop-project.ps1` 负责安全停止当前项目占用的端口服务。

### 2. 启动前检查

启动脚本在真正拉起服务前进行以下检查：

- 检查项目根目录是否正确。
- 检查 `.venv311\Scripts\python.exe` 是否存在。
- 检查管理端和用户端的 `package.json` 是否存在。
- 检查 Node 是否可用，并输出当前版本。
- 检查 8080、8081、8082 端口占用情况。

处理策略：

- 若端口未被占用，则正常启动对应服务。
- 若端口已被当前项目服务占用，则提示“已运行”，不重复启动。
- 若端口被其他未知进程占用，则明确提示端口冲突，并给出停止建议，不强行覆盖。

### 3. 后端启动策略

后端固定使用虚拟环境解释器：

- `.\.venv311\Scripts\python.exe .\run.py`

不再依赖：

- 全局 `python`
- 全局 `python3`
- 旧的 `manage.py run`

这样可以避免：

- 缺少 `pymysql`
- 装到错误 Python 版本
- 使用错误 Flask 版本

### 4. Flask 兼容策略

后端兼容保留以下约束：

- 若运行环境为 Flask 2.x，则允许使用 `json provider` 写法。
- 若运行环境为 Flask 1.1.2，则自动回退到 `json_encoder` 写法。

这部分兼容逻辑直接保留在：

- `api/__init__.py`

目标是保证当前项目在既有依赖不升级的情况下也能稳定启动。

### 5. 前端启动策略

管理端和用户端分别在各自目录执行：

- 管理端：`npm run serve`
- 用户端：`npm run serve`

启动脚本将分别启动两个独立进程，并将输出重定向到项目内日志文件，避免命令行窗口关闭后难以排查。

### 6. 日志与状态输出

启动脚本应在成功后明确输出：

- 后端地址：`http://localhost:8080`
- 管理端地址：`http://localhost:8081/#/login`
- 用户端地址：`http://localhost:8082`
- 默认管理员账号：`admin / admin`

同时输出日志路径：

- 后端日志
- 后端错误日志
- 管理端日志
- 用户端日志

这样用户即使忘记命令，也能快速定位问题。

### 7. 停止脚本策略

`stop-project.ps1` 只处理本项目约定端口：

- `8080`
- `8081`
- `8082`

策略：

- 先查询监听端口对应 PID。
- 找到后提示进程信息并执行停止。
- 只按端口停，不做更宽泛的全局 `node/python` 杀进程。

目标是尽量避免误伤其他项目。

### 8. 文档修复

需要更新以下文档/脚本内容：

- `说明文档.md`
- `readme.md`
- `run.bat`

文档中需要明确：

- 当前项目推荐启动方式
- 当前真实端口
- 需要的 Python/Node 环境
- 常见启动问题与对应日志位置

## 用户体验预期

完成后，用户只需要执行以下任一方式即可：

```powershell
.\start-project.ps1
```

或：

```powershell
.\start-project.bat
```

脚本将自动：

- 检查环境
- 拉起后端
- 拉起管理端
- 拉起用户端
- 打印可访问地址

## 风险与处理

### 风险 1：端口被其他程序占用

处理：

- 启动前检测端口
- 不强行覆盖
- 给出明确冲突提示

### 风险 2：用户切回了错误 Node 版本

处理：

- 启动脚本打印当前 Node 版本
- 文档写明推荐 Node 16
- 若检测到明显异常版本，可在启动输出中给出提醒

### 风险 3：虚拟环境缺失或损坏

处理：

- 启动脚本先检查 `.venv311`
- 若不存在，停止继续执行并提示用户重新安装依赖

### 风险 4：旧脚本继续被使用

处理：

- 让 `run.bat` 转发到新入口
- 把旧命令从文档中移除

## 验收标准

满足以下条件视为方案完成：

1. 在项目根目录执行统一启动脚本后，后端、管理端、用户端均能启动。
2. 访问 `8080`、`8081`、`8082` 时均有正确响应。
3. 管理端可进入登录页。
4. 文档中的启动命令和真实启动方式一致。
5. `run.bat` 不再使用过期命令。
6. 后端不会再次因为 Flask 1.1.2 / 2.x JSON 写法不兼容而直接报错。

## 实施范围

本次会修改以下类型文件：

- 启动脚本：新增/更新
- 文档说明：新增/更新
- 后端兼容文件：小范围修复

不会修改：

- 数据库结构
- 前后端业务功能
- 生产部署配置
